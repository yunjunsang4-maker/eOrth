# Aurora 지구본 확대 시 구분선·육지 정렬 개선 — 설계 문서

- 날짜: 2026-07-22
- 브랜치: feat/empty-social-tab (작업은 별도 브랜치 권장)
- 대상 파일: `src/components/GlobeView.tsx` (Aurora / Neon Globe HTML 문자열 내부만)
- 형태: form1 = Aurora(정사영 납작 원반, `neonGlobeHTML`). Classic은 대상 아님.

## 문제 정의

색으로 활성화되는 Aurora 지구본을 확대하면 두 증상이 나타난다.

1. **선이 육지 경계와 어긋남** — 국경 구분선이 색칠된 육지/해안선 가장자리에서 바깥으로 떠 보이거나 겹쳐 보인다.
2. **육지(색면)가 모자이크됨** — 확대 시 색면 가장자리가 계단·뭉갬 현상을 보인다.

두 증상은 "육지 = 래스터 텍스처, 국경선 = 별도 벡터"라는 이중 구조에서 비롯된다.

## 현재 구조 (Aurora 반지름 스택)

| 레이어 | 반지름 R | 정체 | 렌더 |
|--------|---------|------|------|
| 본체 텍스처 (셰이더 `uLand`) | 1.0 | 색면(육지) | 래스터 (Canvas→CanvasTexture) |
| 딥줌 지역창 오버레이 (`regionMesh`) | 1.0006 | 고해상 재투영 채움 | 래스터 (3072 캔버스) |
| admin1 (주/도선) | 1.001 | 벡터 | LineSegments |
| 국경 110m / 50m | 1.002 | 벡터 | LineSegments |
| 국경 10m | 1.0025 | 벡터 | LineSegments |

관련 함수: `buildNeonTexture()`(전역 텍스처), `buildRegionTexture()`(지역창), `updateRegion()`/`updateRegionFade()`(지역창 활성/페이드), `updateVectorLines()`(벡터 국경 3단계 페이드), `buildWorldLinesMerged()`/`buildPolylinesMerged()`(벡터 라인 생성).

## 원인 진단 (증상 2개 = 뿌리 3개)

1. **모자이크 육지** — 전역 텍스처는 4096 고정. 고해상 지역창은 `REGION_AT*0.9 = 4.5`배부터 활성(`updateRegion()`의 `z<REGION_AT*0.9` 가드). 그러나 벡터 국경선은 1.35배부터 선명(`updateVectorLines()`의 `smoothstep01(1.35, 2.0, z)`). 결과적으로 **줌 2.6~4.5 구간**에서 육지는 늘어난 4096 텍스처(모자이크)인데 선만 쨍해 대비가 최대가 된다.
2. **선–육지 시차** — 색면은 R=1.0(전역) 또는 1.0006(지역창), 국경선은 R=1.002~1.0025. 정사영에서 화면 오프셋 = `(R_선 − R_육지) × 화면중심거리 × 줌배율`. 이 0.2% 반지름 차가 줌·중심거리에 비례해 "바깥으로 밀린 선"으로 보인다.
3. **일반화 레벨 불일치** — 채움(전역=110m/50m, 지역창=10m 마스크)과 그 시점에 보이는 벡터선(110m→50m→10m)의 원본 해상도가 달라, 전환 구간에서 해안선이 다르게 그려져 겹친다.

## 채택 방향

**A안(정렬 & 지역창 조기 활성, 래스터 유지) 우선 적용. 딥줌 극단에서 잔여 시차가 남으면 C 하이브리드(딥줌 구간에 한해 국경선을 지역 텍스처에 함께 굽기)를 그 구간에만 얹는다.**

B안(육지를 벡터 채움으로 통일)은 근본 해결이지만 earcut 삼각분할·국가별 색·iOS 메모리·유리 셰이더 룩 훼손 리스크가 커 A+C로 불충분할 때만 재검토한다.

아래 수치는 **Fable + 루프 스킬로 눈으로 맞추는 튜닝의 출발점**이며, 최종 값은 실기기 육안 확인으로 확정한다.

## A안 변경 상세

### 변경 1 — 벡터 국경선 반지름 정렬 (시차 제거)

`updateVectorLines()`에서 각 벡터 그룹 생성 시 넘기는 반지름을 낮춘다.

- 국경 110m/50m: `1.002 → 1.0008` (`buildWorldLinesMerged(..., 1.002)` 2곳)
- 국경 10m: `1.0025 → 1.0009` (`buildPolylinesMerged(borders10Lines, 1.0025)`)
- admin1(주/도): `1.001 → 1.0008` (`buildPolylinesMerged(admin1Lines, 1.001)`)
- 지역창 오버레이 `1.0006`, 본체 `1.0`은 **유지**

근거: 선은 `depthWrite:false`와 `renderOrder`로 그리기 순서가 이미 위이므로 반지름을 낮춰도 육지 위에 계속 얹힌다. 딥줌(육지=지역창 1.0006)에서 (R_선−R_육지)를 0.002 → ~0.0002로 줄여 시차를 약 10배 감소.

리스크/가드: 본체(1.0)와 너무 가까우면 z-파이팅. 1.0008은 지역창 1.0006이 이미 검증된 대역 안이라 안전선으로 본다. 육안 확인에서 깜빡임이 있으면 1.0008~1.0012 범위에서 상향 조정.

### 변경 2 — 지역창 조기 활성 (모자이크 밴드 제거)

- `REGION_AT: 5 → 3.5`
  - 활성 가드 `z<REGION_AT*0.9` → 줌 ~3.15부터 지역창이 채움을 이어받음
  - land10m 요청 가드 `z>REGION_AT*0.7` → 데이터 lazy 요청도 함께 앞당겨짐
  - span = `Math.max(2.5, Math.min(60, 480/z))`, S=3072 → 60°에서도 전역(4096/360≈11px/°) 대비 51px/°로 ~4.6× 선명
- 전역 텍스처 4096은 **그대로 유지** (8192는 iOS 캔버스 메모리로 빈 텍스처→활성색 꺼지던 함정)
- 지역창 재생성은 `zoomSettled()` 게이트가 이미 있어 확대 중 렉 없음

리스크/가드: 조기 활성으로 지역창 재생성 빈도가 늘 수 있으나 단일 재사용 캔버스(`window.__regionCv`) + settle 게이트로 완화됨. 날짜변경선 창은 기존대로 전역 폴백 유지(`Math.abs(c.lon)>180-span`).

### 변경 3 — 일반화 레벨 통일 (해안선 겹침 제거)

지역창 채움은 10m 마스크(`land10`)인데 그 시점 벡터선이 50m면 해안선이 어긋난다.

- `borders10` 페이드인 임계값을 지역창 활성과 맞물리게 앞당김: `smoothstep01(4.5, 5.8, z) → smoothstep01(3.5, 4.5, z)`
- 지역창이 켜지는 순간부터 10m 벡터가 주선이 되고 50m 벡터는 페이드아웃 → 채움 마스크(10m)와 선(10m)이 같은 원본 해상도

리스크/가드: 10m 데이터 도착 전(요청~응답 지연)에는 기존 50m 벡터로 자연 폴백(현재 `borders10Lines` null 가드 유지). 시각적으로는 페이드라 급전환 없음.

## 데이터 흐름 (변화 없음)

`needLand10m` / `needBorders10m` / `need50m` / `needAdmin1` lazy 요청 파이프라인(WebView→RN)은 구조 변경 없음. 임계값만 앞당겨져 10m 계열 데이터를 조금 더 일찍 요청한다.

## C 하이브리드 (A로 딥줌 극단이 부족할 때만)

가장 깊은 줌에서 잔여 시차가 남을 경우에 한해:

- `buildRegionTexture()`에서 `destination-in`(10m 마스크) **이후** `source-over`로 10m 국경선 흰 스트로크를 지역 캔버스에 함께 굽는다.
- 해당 딥줌 구간에서는 `borders10Group` 벡터선을 페이드아웃한다.
- 결과: 채움 + 선이 한 투영·한 반지름의 래스터가 되어 완전 일치. 지역 캔버스가 고해상(3072/좁은 span)이라 선 선명도 유지.

A 적용 후 육안 판단으로 도입 여부 결정. (선행 조건: A안 변경 1~3 완료 및 검증)
